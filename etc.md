# ✅ ОБНОВЛЁННЫЙ CURSOR PROMPT

## Task 0.1 — Создание Django-проекта (task-oriented architecture)

Скопируй **целиком**, без правок 👇

---

> Создай Django-проект с использованием Django Ninja.
>
> Архитектура проекта должна быть ориентирована на разные типы заданий ЕГЭ,
> где **каждый тип задания — отдельный Django app**.
>
> ### Общие требования
>
> * Все приложения должны находиться в папке `apps/`
> * Использовать SQLite как базу данных по умолчанию
> * Подключить Django Ninja и базовый API router
> * Проект должен запускаться сразу после создания
> * Код должен быть аккуратным, расширяемым и production-ready
>
> ### Обязательные apps
>
> Создай следующие Django apps:
>
> * `users` — кастомная модель пользователя
> * `attempts` — универсальные попытки решений
> * `core` — базовые абстракции и общие интерфейсы
> * `ai_engine` — LangChain, LLM, prompts, chains
>
> ### Task-ориентированные apps
>
> Для каждого типа задания создать отдельный app с префиксом `task_`
> (например: `task_truth_table`, `task_code_exec`).
>
> **ВНУТРИ КАЖДОГО `task_xxx` app строго использовать следующую структуру:**
>
> ```
> task_xxx/
> ├── models.py     # только модели, специфичные для этого типа задания
> ├── services.py   # чистая бизнес-логика и алгоритмы решения (без Django)
> ├── ai.py         # prompts и LangChain chains только для этого типа задания
> ├── views.py      # Django Ninja endpoints (тонкий слой)
> ├── admin.py
> ├── apps.py
> └── __init__.py
> ```
>
> ### Архитектурные правила (обязательные)
>
> * `views.py` **НЕ содержит бизнес-логики**
> * `services.py` **НЕ зависит от Django**
> * `ai.py` **НЕ содержит HTTP-логики**
> * Общие абстракции задач и попыток размещать в `core`
> * Связь `attempts` ↔ `task_xxx` реализовать через `GenericForeignKey`
>
> ### Settings
>
> * Вынести настройки в `config/settings/base.py` и `config/settings/dev.py`
> * Подключить все apps в `INSTALLED_APPS`
>
> ### Результат
>
> * Проект успешно запускается (`python manage.py runserver`)
> * Структура готова к добавлению новых типов задач без рефакторинга

---

## 🧠 Почему этот промт «работает» с Cursor

* 🔒 жёстко зафиксирована структура `task_xxx`
* 🧩 разделена ответственность файлов
* 🛑 прямо запрещена бизнес-логика во `views.py`
* 📐 Cursor не «угадывает», а следует правилам

# ✅ CURSOR PROMPT

## Task 0.2 — Core abstractions (foundation layer)

Скопируй **целиком** 👇

---

> Реализуй **core-слой** Django-проекта, который задаёт общие абстракции
> для всех типов заданий ЕГЭ и попыток их решения.
>
> Core должен быть **тонким, стабильным и расширяемым**, без логики конкретных заданий.
>
> ---
>
> ### Структура `apps/core/`
>
> ```
> core/
> ├── models.py        # базовые абстрактные модели
> ├── interfaces.py    # протоколы / интерфейсы сервисов
> ├── enums.py         # общие перечисления
> ├── exceptions.py   # доменные исключения
> ├── utils.py        # вспомогательные функции (минимум)
> └── __init__.py
> ```
>
> ---
>
> ### 1. BaseTask (абстрактная модель)
>
> Создай абстрактную Django-модель `BaseTask`, которая будет
> родительской для всех `task_xxx.models`.
>
> Требования:
>
> * `id`
> * `title`
> * `description`
> * `difficulty` (enum)
> * `created_at`
> * `updated_at`
> * `is_active`
> * `Meta.abstract = True`
>
> BaseTask **НЕ содержит**:
>
> * логики решения
> * AI
> * попыток
>
> ---
>
> ### 2. BaseAttempt (абстрактная модель)
>
> Создай абстрактную модель `BaseAttempt`:
>
> Поля:
>
> * `user` (FK)
> * `is_correct`
> * `created_at`
> * `confidence_score` (float, nullable)
>
> BaseAttempt **НЕ знает**, к какому типу задания относится.
>
> ---
>
> ### 3. Универсальная Attempt (apps/attempts)
>
> Подготовь контракт, чтобы `attempts.Attempt`:
>
> * наследовался от `BaseAttempt`
> * был связан с любой задачей через `GenericForeignKey`
>
> Core должен содержать:
>
> * абстракции и типы, необходимые для этой связи
>
> ---
>
> ### 4. Интерфейс сервиса решения задачи
>
> В `core/interfaces.py` определи интерфейс `TaskServiceInterface`.
>
> Он должен описывать **чистую бизнес-логику**:
>
> ```python
> solve(task, user_input) -> SolveResult
> check(task, user_input) -> bool
> ```
>
> Интерфейс:
>
> * не зависит от Django
> * не зависит от AI
>
> ---
>
> ### 5. Интерфейс AI для задач
>
> В `core/interfaces.py` добавь интерфейс `AITaskInterface`:
>
> ```python
> analyze(task, attempt) -> AIAnalysis
> hint(task, level) -> Hint
> explain(task) -> str
> ```
>
> AI-интерфейс:
>
> * не знает про HTTP
> * не знает про ORM
> * используется внутри `task_xxx.ai`
>
> ---
>
> ### 6. Общие enum’ы
>
> В `core/enums.py`:
>
> * `DifficultyLevel` (EASY / MEDIUM / HARD)
> * `AttemptStatus` (CORRECT / WRONG / PARTIAL)
>
> Использовать `TextChoices`.
>
> ---
>
> ### 7. Доменные исключения
>
> В `core/exceptions.py`:
>
> * `TaskValidationError`
> * `SolveError`
> * `AIError`
>
> Использовать их в сервисах, не в views.
>
> ---
>
> ### 8. Архитектурные ограничения (ОБЯЗАТЕЛЬНО)
>
> * `core` **НЕ импортирует**:
>
>   * `task_xxx`
>   * `ai_engine`
>   * `views`
> * `core` — самый нижний слой
> * Любой `task_xxx.services` обязан реализовывать `TaskServiceInterface`
> * Любой `task_xxx.ai` обязан реализовывать `AITaskInterface`
>
> ---
>
> ### Результат
>
> * Core используется всеми task-apps
> * Добавление нового `task_xxx` не требует изменения core
> * Код чистый, с type hints и docstrings
