## 2️⃣ Расчёт нагрузки

### 2.1. Пользовательская активность

**Год 1: 10 000 пользователей**

* Активные одновременно (peak) ≈ 5–10% → 500–1 000 concurrent users
* Среднее количество LLM запросов в секунду:

[
\text{LLM qps} = \frac{\text{пользователи одновременно} \times \text{задания/сессия}}{\text{средняя продолжительность сессии в секундах}}
]

Берём:

* 1 000 concurrent users
* 15 заданий/сессия
* 40 минут/сессия = 2 400 секунд

[
\text{LLM qps} \approx \frac{1 000 \times 15}{2 400} \approx 6.25 \text{ запросов/сек}
]

То есть **≈6–7 QPS на LLM** в пиковое время.

---

**Год 2: 100 000 пользователей**

* Активные одновременно (peak) ≈ 5% → 5 000 concurrent users
* LLM qps:

[
\text{LLM qps} \approx \frac{5 000 \times 15}{2 400} \approx 31.25 \text{ запросов/сек}
]

> Плюс запас на рост и пиковые часы → рассчитываем на **≈40 QPS**.

### 2.2. База данных

* Таблицы: `users`, `tasks`, `attempts`, `task_xxx_models`
* Средний размер attempt (JSON) ≈ 2 KB
* Количество attempts/год:

[
10 \text{ тыс. юзеров} \times 200 заданий/год \approx 2 \text{ млн записей}
]

* Для 100 тыс пользователей → 20 млн записей

SQLite **не подойдёт на 2-й год** → сразу планируем **PostgreSQL / CockroachDB / Aurora**.

## 3️⃣ Серверная архитектура

### 3.1. Backend

* Django + Ninja API
* Масштабирование через **Gunicorn + Uvicorn + Nginx**
* Первичный год: 2–3 сервера (2 CPU, 8 GB RAM) → load balancing
* Второй год: 6–8 серверов (4 CPU, 16 GB RAM)

### 3.2. LLM (OpenAI / Claude)

* Рекомендуем **отдельные сервисы / пул**:

| Год | QPS | Модель       | Пулы                         |
| --- | --- | ------------ | ---------------------------- |
| 1   | 6–7 | GPT-4.1-mini | 1–2 параллельных экземпляра  |
| 2   | 40  | GPT-4.1-mini | 4–6 параллельных экземпляров |
| Пик | 50  | GPT-4.1-mini | 6–8 параллельных экземпляров |

> Каждое обращение к LLM ~ 1–2 секунд latency → параллельные инстансы снижают задержку.

---

### 3.3. Кэширование

* Redis / Memcached для:

  * Часто используемых промтов (шаблоны задач, подсказки)
  * Сессий пользователей
  * Кэш AI-анализов, чтобы не дергать LLM повторно

---

### 3.4. Хранилище данных

* PostgreSQL / Aurora: attempts, users, задачи
* S3 / MinIO: медиа и файлы (если будут решения с картинками или диаграммами)

---

### 3.5. Архитектура по слоям

```
Frontend (React / HTMX)
       │
       ▼
API Layer (Django Ninja)
       │
       ▼
Services Layer:
  - Task services (business logic)
  - AI Engine (LangChain + LLM)
       │
       ▼
Databases:
  - PostgreSQL / Aurora
  - Redis (cache)
  - Object storage (S3)
```

## 5️⃣ Рекомендации по масштабированию

1. **Сразу использовать PostgreSQL**, SQLite только для dev.
2. **LLM-инстансы в пуле**, чтобы выдерживать пик 50 QPS.
3. **Redis кэш** для AI-ответов и шаблонов задач → экономия денег на API.
4. **Разделять AI и бизнес-логику** → можно horizontal scaling отдельно.
5. **Мониторинг:** Prometheus + Grafana, чтобы отслеживать:

   * latency LLM
   * qps API
   * DB connections
6. **Бэкапы и репликация** базы данных для безопасности.
